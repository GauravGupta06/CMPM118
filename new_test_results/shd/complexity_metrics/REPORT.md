# SHD Input-Complexity Router Sweep

Generated: 2026-05-03

## Question

Can we separate "easy" and "hard" SHD samples without training a learned router, using a deterministic metric that could later be ported to STM32 and measured for real energy cost?

For this sweep, "hard" means:

```text
dense_needed = dense_correct and not sparse_correct
```

That is the exact routing-positive case where sending the sample to the dense model can recover an error made by the sparse model.

## Research Basis

- SHD is not a static image-style dataset. It is an audio-derived spike dataset with 20 labels: spoken digits 0-9 in English and German. The official SHD resource says the samples are generated by an artificial cochlea into 700 input channels and include 2264 test samples: https://zenkelab.org/resources/spiking-heidelberg-datasets-shd/
- The original Heidelberg paper emphasizes that SHD has variable audio duration, speaker variation, held-out speakers in test, and that spike timing is essential for accuracy, so a real per-sample complexity distinction is plausible: https://www.kip.uni-heidelberg.de/Veroeffentlichungen/download.php/6616/temp/4143-3.pdf
- The SHD baseline we are matching uses 700 input channels, 512 hidden neurons for SHD, 1 ms timestep, 14 Hz target hidden firing rate, and reports 0.42 mJ for the no-delay feed-forward Loihi 2 model: https://arxiv.org/pdf/2510.13757
- Entropy-based spike-train measures are defensible because spike timing can carry information independently of rate. A classic reference is Strong et al., "Entropy and information in neural spike trains": https://collaborate.princeton.edu/en/publications/entropy-and-information-in-neural-spike-trains/
- Lempel-Ziv complexity is a known spike-train complexity/entropy-rate proxy, but it must be benchmarked for energy because it is more expensive than simple counters: https://pubmed.ncbi.nlm.nih.gov/15025827/
- Event-stream literature also supports event-count, timing, entropy, and adaptive slicing ideas as meaningful descriptors of event-stream difficulty/information content: https://pmc.ncbi.nlm.nih.gov/articles/PMC9003192/

## Implemented File

`evaluate_shd_complexity_metrics.py`

The script evaluates many deterministic, non-learned metrics in one file:

- Activity/load: total spikes, spike density, active frames, active channels, early prefix spike counts.
- Temporal structure: duration, centroid, spread, entropy, HHI, Gini, total variation, active/inactive transitions, burst count, longest silence.
- Cochlear/channel structure: channel centroid, channel spread, channel entropy, channel HHI, channel total variation, low/high channel ratio.
- LZC variants: legacy LZC score, active-time LZC, active-channel LZC, early prefix LZC.

No new router model is trained. The script reports oriented ROC AUC, threshold, route split, routed accuracy, and model-only energy savings. It intentionally does not estimate STM32 energy for the new metrics; that must be measured on hardware.

## Separability Result

There is a real distinction, but it is modest rather than clean. The dense-needed class is only 222 / 2264 samples, or 9.81% of the test set. This makes routing hard: most samples do not need dense, and many scalar metrics only weakly identify the few that do.

## Dataset Structure Observed

Dense-needed samples are not evenly distributed across labels. Highest dense-needed rates:

| Label | Word | Dense-needed |
|---:|---|---:|
| 13 | drei | 38 / 120 = 31.7% |
| 15 | fuenf | 29 / 112 = 25.9% |
| 17 | sieben | 31 / 121 = 25.6% |
| 1 | one | 20 / 108 = 18.5% |
| 9 | nine | 16 / 103 = 15.5% |
| 10 | null | 17 / 122 = 13.9% |

Some labels had essentially no dense-needed cases:

```text
eight: 0 / 114
acht:  0 / 123
six:   1 / 109
two:   1 / 105
```

So yes, the dataset/model pair has "harder" and "easier" regions, but the distinction is partly semantic/class-driven and partly input-statistic-driven. Since the router does not know the true label before inference, we need metrics that can infer those regions from the event stream itself.

## Interpretation

The best signals are mostly load and temporal-distribution measures. Dense-needed samples tend to have more total spikes and more spikes in the first 400 ms. They also tend to have lower temporal concentration metrics, meaning activity is spread in a way that the sparse model may not represent as reliably. That makes intuitive sense: the sparse model has roughly half the spike budget of the dense model, so the cases where it loses are often the cases with more evidence to integrate across time/frequency channels, not just "random-looking" spike trains.

The current AUC ceiling around 0.65 is good enough to prove a distinction exists, but not good enough for a final research-paper router by itself. A single cheap scalar can recover accuracy close to dense while saving model energy, but once router energy is included, the metric must be extremely cheap. The best candidate right now is not LZC; it is a prefix event-count family metric, especially `prefix_400ms_spikes`, because it is just an integer counter, easy to implement in C, and likely very cheap to measure on STM32.

## Recommended Next Step

Measure STM32 energy for a very small shortlist:

1. `prefix_400ms_spikes`
2. `burst_count`
3. `active_frame_transitions`
4. `temporal_hhi`
5. `time_active_lzc`

Then use the cheapest metric that still gives acceptable routed accuracy. My current bet is that a prefix-spike-count router will beat LZC on total energy even if LZC has comparable accuracy, because it is dramatically simpler: one counter and one threshold.
